# Importing Cloudflare D1 into PlanetScale Postgres

This document describes how `pscale import d1` works on the `import-d1` branch: the end-to-end flow from a wrangler `.sql` export to a loaded PlanetScale Postgres branch, plus features and edge cases we do **not** handle today.

For machine-readable output, add `--format json` to any command below.

## Prerequisites

- **PlanetScale CLI** built from this branch (`go build -o bin/pscale ./cmd/pscale`)
- **PlanetScale auth** â€” `pscale auth login` and `pscale org switch <org>`
- **Target database** â€” a **PostgreSQL** database branch (MySQL/Vitess branches are rejected)
- **Tools** (checked by `pscale import d1 doctor`):
  - **pgloader** â€” required for data load
  - **psql** â€” required for schema/index application (small-dump path)
  - **sqlite3** â€” required for staging and verify (optional in doctor, required at import time)
  - **wrangler** â€” optional if you already have a `.sql` export

## High-level flow

```
wrangler d1 export â†’ .sql file
        â†“
   doctor / lint / start --dry-run   (read .sql only; no Postgres writes)
        â†“
   start (real import)
        â”śâ”€ build staging .sqlite from .sql
        â”śâ”€ convert schema â†’ Postgres DDL
        â”śâ”€ pgloader: sqlite â†’ Postgres (data)
        â””â”€ apply indexes after load
        â†“
   verify â†’ complete
```

## Step-by-step

### 0. Export from D1 (Cloudflare)

```bash
wrangler d1 export <database-name> --remote --output ./d1-export.sql
```

This produces a SQLite-format SQL dump (schema + `INSERT` data), the same shape wrangler uses for D1 backups.

The CLI does **not** run wrangler for you today; you pass the file with `--input`.

---

### 1. `pscale import d1 doctor`

Checks local prerequisites: wrangler, pgloader, psql, sqlite3, and Cloudflare env vars (`CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`).

- **Does not** read your dump or touch PlanetScale.
- Fails import if pgloader or psql is missing.

---

### 2. `pscale import d1 lint --input ./d1-export.sql`

Static analysis of the **`.sql` file`** (stream-parsed; no SQLite file created).

**What it inspects:**

| Area | Examples |
|------|----------|
| Identifiers | Names over Postgres 63-byte limit (error); mixed-case names (warning) |
| Types / coercion hints | `AUTOINCREMENT`, integer columns sampled as 0/1 (boolean), `*_at` TEXT â†’ timestamptz, JSON-like TEXT â†’ jsonb, UUID-shaped TEXT |
| ORM migration tables | Drizzle, Prisma, Rails, Knex, etc. (info â€” these are skipped on import) |
| Virtual tables | `CREATE VIRTUAL TABLE` (FTS5, etc.) â€” **parse error**, blocks import |

**Blocking behavior:** lint **errors** (currently mainly `IDENTIFIER_TOO_LONG`) block `start`. Warnings and info do not.

---

### 3. Plan (inside `start`, not a separate command)

There is no standalone `pscale import d1 plan` command. Planning runs automatically as part of **`start`** (including `--dry-run`).

**What plan does** (still reads `.sql` only):

1. Parse all `CREATE TABLE` statements
2. Count rows per table from `INSERT` statements
3. Measure dump file size
4. Recommend load method:
   - **`psql`** â€” dumps under ~1 GB (schema via psql, data still via pgloader)
   - **`pgloader`** â€” larger dumps
5. Compute **load order** â€” topological sort using foreign keys (parents before children)
6. Attach default **pgloader cast rules** (boolean columns, timestamptz, jsonb)
7. Assign a **`migration_id`** and save **local state** (`phase: planned`) under your PlanetScale config directory

**Preview without loading:**

```bash
pscale import d1 start <database> [branch] \
  --input ./d1-export.sql \
  --dry-run \
  --format json
```

Returns lint results + plan (tables, method, migration_id). **No Postgres connection required for dry-run.**

---

### 4. `pscale import d1 convert-schema` (optional)

```bash
pscale import d1 convert-schema --input ./d1-export.sql --output ./postgres-schema.sql
```

Converts SQLite DDL from the dump to Postgres `CREATE TABLE` + `CREATE INDEX` SQL. Useful for inspection or manual edits. The real import runs conversion internally; you do not need this step for `start`.

**Conversion behavior:**

- SQLite types mapped to Postgres (e.g. `INTEGER` â†’ `BIGINT`, `BLOB` â†’ `BYTEA`, sampled booleans â†’ `BOOLEAN`)
- `AUTOINCREMENT` â†’ `GENERATED BY DEFAULT AS IDENTITY`
- Table-level `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `CHECK` constraints preserved when recognized
- Secondary indexes extracted from `CREATE INDEX` in the dump
- ORM metadata tables omitted from output

---

### 5. `pscale import d1 start` (real import)

```bash
pscale import d1 start <database> [branch] \
  --input ./d1-export.sql \
  --migration-id <id-from-dry-run> \
  --force \
  --format json
```

Positional args: `<database> [branch]` (default branch: `main`). Org comes from `pscale` config.

**Phases:**

1. **Lint + plan** (same as dry-run)
2. **SQLite staging** â€” builds `<dump-basename>.sqlite` next to the `.sql` file (e.g. `d1-export.sqlite`) by feeding the dump to `sqlite3` in chunks. Reused on retry if dump size/mtime unchanged.
3. **Schema** â€” converted Postgres DDL applied via `psql` (tables only; indexes deferred)
4. **Data** â€” pgloader loads from `sqlite:///<staging.sqlite>` into PlanetScale Postgres, one table at a time for large DBs (resume-friendly)
5. **Indexes** â€” secondary indexes applied after data load (faster than maintaining indexes during insert)

**Flags:**

| Flag | Purpose |
|------|---------|
| `--dry-run` | Lint + plan only |
| `--migration-id` | Resume/reuse a prior dry-run plan |
| `--method pgloader\|psql` | Override auto method selection |
| `--force` | Skip human confirmation prompt |

**Requires:** authenticated PlanetScale client, PostgreSQL target branch, pgloader + psql + sqlite3.

---

### 6. `pscale import d1 status`

Shows local migration state for a `--migration-id`: phase, input path, method, loaded tables, etc. Suggests the next command based on phase.

**Migration phases:**

| Phase | Meaning |
|-------|---------|
| `planned` | Dry-run completed; ready to import |
| `importing` | Import in progress |
| `imported` | Data loaded; run verify |
| `verified` | Verify passed; run complete |
| `complete` | Marked done |
| `failed` | Import failed; can retry with same migration id |

---

### 7. `pscale import d1 verify`

```bash
pscale import d1 verify <database> [branch] \
  --migration-id <id> \
  --input ./d1-export.sql \
  --format json
```

Compares source (staging SQLite) to destination (PlanetScale Postgres).

**Checks:**

- Row counts per table (source vs dest)
- Identity/sequence values for `AUTOINCREMENT` columns
- Boolean column distributions (0/1 columns converted to boolean)
- Table fingerprints (checksum-style aggregates)
- Sample row content (tables with a single-column primary key; up to 8 tables, 3 rows each)

On success, updates migration state to `verified`.

---

### 8. `pscale import d1 complete`

```bash
pscale import d1 complete <database> [branch] --migration-id <id> --force
```

Marks migration `complete`. Requires `verified` phase. Reminds you to **re-baseline ORM migration history** on Postgres (migration metadata tables from D1 were not imported).

---

## Why we build a `.sqlite` file

A wrangler export is a **text script**, not a queryable database.

| Step | Reads `.sql` | Reads `.sqlite` |
|------|--------------|-----------------|
| lint / plan | âś“ | |
| import (pgloader) | | âś“ (pgloader source is `sqlite:///â€¦`) |
| verify | âś“ (parse) | âś“ (COUNT/SELECT queries) |

Staging runs the dump through `sqlite3` once so downstream tools see real SQLite tables and types. The staging file is cached beside the dump for faster retry/verify.

---

## What we intentionally skip

### ORM / framework migration tables

These tables are **not imported** into Postgres (data-only path skips them; schema conversion omits them):

- Drizzle (`__drizzle_*`)
- Prisma (`_prisma_migrations`)
- Rails (`schema_migrations`, `ar_internal_metadata`)
- Knex, Sequelize, Flyway, Liquibase, Django, Alembic, TypeORM, Goose, â€¦

After import you must **re-baseline** your ORM on Postgres. `complete` surfaces detected ORMs and remediation hints.

## Edge cases and unsupported features

### Hard failures (import blocked or parse error)

| Feature | Behavior |
|---------|----------|
| **FTS5 / virtual tables** | `CREATE VIRTUAL TABLE` fails parse with `VIRTUAL_TABLE` â€” remove or recreate manually in Postgres (e.g. full-text search) |
| **Identifier > 63 bytes** | Lint error; blocks import |
| **Non-PostgreSQL target** | Import rejected if database kind is not `postgresql` |
| **Destination table name conflict** | Fails if target branch already has overlapping table names from a prior import |
| **Lint errors** | Block `start` until fixed |

### Not migrated (silently ignored or lost)

| Feature | Behavior |
|---------|----------|
| **Triggers** | **Lint warning** (`TRIGGER_NOT_MIGRATED`) â€” may exist in staging SQLite only; never created in Postgres |
| **Views** | **Lint warning** (`VIEW_NOT_MIGRATED`) â€” not parsed or converted; recreate manually in Postgres |
| **Partial indexes** | **Lint warning** (`PARTIAL_INDEX`) â€” `WHERE` clause is not preserved; a non-partial index may be created instead |
| **Expression indexes** | **Lint warning** (`EXPRESSION_INDEX`) â€” non-simple index DDL is not migrated |
| **SQLite-specific pragmas** | `PRAGMA` lines are executed during SQLite staging but have **no Postgres equivalent** in our pipeline |
| **ATTACH DATABASE** | Not supported |
| **WITHOUT ROWID / STRICT tables** | Not specially handled; may not map cleanly to Postgres |
| **Custom SQLite functions** | Used in triggers/views/checks â€” not ported |
| **D1-specific extensions** | Anything beyond standard SQLite wrangler export shape is untested |

### Heuristic type coercion (may surprise you)

Type mapping uses **column names + sampled INSERT values**:

- Integer columns with only `0`/`1` samples â†’ `BOOLEAN`
- Column names like `is_*`, `*_flag`, `active`, `enabled` â†’ boolean hints (with sample validation)
- `*_at`, `timestamp`, `date` TEXT columns â†’ `TIMESTAMPTZ` when samples look like timestamps
- `json`, `metadata`, `payload` TEXT â†’ `JSONB` when samples parse as JSON

**Risk:** columns that look boolean/timestamp/json by name but store other data may be coerced incorrectly, or left as TEXT when samples do not match. Review lint output and run `verify`.

### Foreign keys and load order

Load order is computed from FK references in the dump. Edge cases:

- Parent table defined **after** child in the dump â€” dependency may be missed if the parent was not yet in the parsed set when the child was processed
- Cross-database or external FK targets â€” not modeled

### Verify limitations

- Only compares tables present in the **source dump** (extra Postgres-only tables are flagged only when migration state tracks loaded tables)
- Sample row compare requires a **single-column primary key**; composite PK tables skip deep row sampling
- Boolean verify expects strict **0/1** semantics; values like `2` may load but fail boolean distribution checks
- BLOB/BYTEA compare uses hex encoding; very large blobs are summarized in output

### Operational limits

- **Offline import** â€” requires a local `.sql` file; no live D1 API sync
- **Single branch target** â€” one migration id tracks one org/database/branch/input path
- **Human confirm** â€” TTY `start` prompts unless `--force` or JSON mode
- **Tooling** â€” pgloader/SBCL memory limits can affect very wide rows or huge tables; large imports load **one table per pgloader invocation** to mitigate this
- **Resume** â€” failed imports can retry with the same `--migration-id`; already-loaded tables may be skipped

## Command cheat sheet

```bash
# Prerequisites
pscale import d1 doctor

# Analyze export
pscale import d1 lint --input ./d1-export.sql --format json

# Preview plan + get migration_id
pscale import d1 start mydb --input ./d1-export.sql --dry-run --format json

# Import
pscale import d1 start mydb --input ./d1-export.sql --migration-id <id> --force --format json

# Post-import
pscale import d1 verify mydb --migration-id <id> --input ./d1-export.sql --format json
pscale import d1 complete mydb --migration-id <id> --force

# Inspect converted schema (optional)
pscale import d1 convert-schema --input ./d1-export.sql --output ./schema.sql
```
